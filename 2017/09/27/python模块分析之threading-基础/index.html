python模块分析之threading-基础

发表于 2017-09-27 | 分类于 python ， python库分析 |

前言

python 的 threading 模块是 python 非常核心的一个模块。它是在 thread 模块基础上进行的封装，里面有许多值得挖掘的点，如多线程编程，线程池，守护线程，线程间阻塞，多进程等相关问题，这里我将对相关知识简单的归纳总结一下

python线程基本介绍

python里面的线程有如下几个特点

1.在CPYTHON中，由于全局解释锁（GIL）的原因，Python 的线程被限制为同一时刻只允许一个线程执行。所以，Python 的线程更适用于处理 I/O 密集型操作和其他需要并发执行的阻塞操作（比如等待从数据库获取数据等等）

2.Python 中的线程是系统级线程（比如说一个 POSIX 线程或者一个 Windows 线程），这些线程由操作系统来管理。线程一旦 start()，将独立执行直到目标函数返回。过程中你可以查询一个线程对象的状态，看它是否还在执行

3.线程划分为用户线程和守护线程 (daemon设置为true)，守护线程级别非常低，虚拟机如果只有守护线程，将会自动退出

守护线程（也可以称为后台线程，服务线程）

先推荐一篇经典文章分析守护线程的文章 http://blog.csdn.net/frankarmstrong/article/details/8880779

守护线程特点如下：

  1. setDaemon 将线程设置为守护进程(也可以在创建线程后设置线程对象的 daemon 属性为 True)

  2. setDaemon 需要在 start 方法调用之前使用

  3. 如果 python 虚拟机中都是守护线程，当前虚拟机将 exit，守护线程级别非常低

  4. 主线程结束后，用户线程将会继续运行，如果没有用户线程，都是后台进程的话，那么虚拟机运行结束

threading 协调线程的启动

在我们系统中如果一个线程的启动运行依赖于另外一个线程，为了协调线程的启动，threading 模块提供了一系列方法来完成，如 Event 模块，Condition 模块等，我们接下来将对常用方法做一些分析

threading.Event 协调线程的启动

官方描述：

核心函数:

  1. Event.set()

  2. Event.wait()

  3. Event.clear()

这段代码非常简单，简单到很难在复杂场景中来复用，守护线程作为了一个定时器，每 5s 通过 Event 下发一个执行任务的令牌，然后允许子进行继续运行，子线程运行完后会清理掉 Event，然后重新等待守护线程分配任务。因为守护线程的任务优先级非常低，因此可以作为 daemon 线程在后台运行。尽管可以通过 clear() 方法来重置 Event 对象，但是很难确保安全地清理 Event 对象并对它重新赋值。很可能会发生错过事件、死锁或者其他问题（特别是，你无法保证重置 Event 对象的代码会在线程再次等待这个 event 对象之前执行）。如果一个线程需要不停地重复使用 Event 对象，最好使用 Condition 对象来代替，为啥呢？

因为 Event 里面是用一个全局变量 self.__flag 作为一个flag来协调线程的启动，实现不够友好，会一次性唤醒所有线程并且没加锁

其它方法

threading.active_count() 获取当前进程的线程数

threading.current_thread().getName() 获取当前线程名称

threading.enumerate() 枚举当前进程的线程对象列表

threading.local() 这是一个非常有特点的对象，他存储的信息都是针对当前线程的，但是这个对象又能被所有线程同时调用，如在线程 A 中 local.name = “张三”，在线程 B 中 local.name = “李四”，在线程 B 中获取name属性，只能获取到李四，相当于这些属性都是线程的私有属性，我们实现这种功能也有其它方式，如通过一个字典，里面名称就是线程名，里面再嵌套一层字典存储线程信息，如 {“线程A”: {“属性A”： 1，”属性B”: 2}}

python框架学习之celery初探

发表于 2016-12-28 | 分类于 python ， python框架学习 |

前言

当我们设计复杂系统的时候经常会需要使用异步任务或者定时任务来处理一些特殊的业务，这时我们就需要一个异步任务调度系统。在 python 里面我们可以使用 celery 配合 rabbitmq（redis）比较容易的实现一个分布式异步任务调度系统

这篇文章主要介绍 celery 的一些基本概念和原理，以及 celery 安装和 coding 实战

celery 简介

官网对celery的解释如下:

celery 是一个依赖于分布式消息队列的异步任务调度系统，是在 redis（或者rabbitmq）上包的一层壳，方便调用，它有四个特点:

1.依赖于其他的消息队列

2.支持异步调用

3.支持分布式

4.支持任务调度（包括实时任务调度和定时任务调度）

补充说明，这里的 redis 只是一个内存数据库，他提供了丰富的数据结构和接口，让我们依赖于他可以实现一个消息队列（celery就是这么干的）

celery 原理介绍

这是我画的一个简易版架构图，从图中可以发现

1.celery 也是一个典型的生产者消费者模型，worker 专门负责处理生产者产生的任务

2.我们可以很容易通过扩充 worker 数目来达到横向扩容的目的

3.因为是基于消息队列（redis或rabbitqm）的，我们可以很方便实现分布式部署 celery（消息队列可以分布式部署，worker也可以分布式部署）

celery 和 redis 以及 rabbitmq 的关系

经常我们说到 celery 都会谈到 redis 和 rabbitmq，celery 难道不是独立的吗，我们通常用到的库或者框架，他都能够正常的工作，celery 是一个例外，他还真不能独立工作，celery 和 redis，rabbitmq 的关系是 redis，rabbitmq 没有了 celery 了依然有其他办法作为消息队列来处理异步任务，但是 celery 没有了 redis、rabbitmq 或者其他消息队列就真没啥用了，既然这样，那为什么还会有 celery 呢？我觉得celery 存在的原因有两点：

1.他对 redis，rabbitmq 等多种消息队列进行了一层统一接口包装，可以替换消息队列而不影响业务系统稳定性

2.对异步任务，实时调度，定时调度等常用操作做了封装，方便开发人员使用

让我联想到了 django 里面的 ORM，也是对数据库操作进行了一次包装和映射，用户不用去关心底层用的什么数据库，甚至都不用去写 sql 代码，依然可以操作数据库，以及对数据库进行随时更换而不影响上层业务代码

celery 有自己的监控组件 flower

安装

安装很简单 pip install flower

如果你想安装开发版 pip install https://github.com/mher/flower/zipball/master#egg=flower

启动

启动 celery 有两种办法

1.直接通过 flower 启动，其中 proj 和之前的 tasks 一样

flower -A proj –port=5555

2.通过 celery 启动，还可以指定绑定端口和 ip，还可以指定 broker –broker=amqp://guest:guest@localhost:5672//

celery flower -A proj –address=127.0.0.1 –port=5555

进阶教程

celery 还支持一些高级特性，如支持注册回掉函数，支持函数注入，更丰富的任务调度特性等

回调函数

因为任务是异步的，实际测试需要在主程序中加入 sleep 才可以等到异步函数返回结果，看到 callback 函数执行 print

函数注入

celery.result.AsyncResult 还支持函数注入，通过函数注入可以和其他进程实现交互，AsyncResult 通过 task_id 找到对应的任务对象，按照之前的 get 方法可以获取函数返回值，其中可以注入 on_message 函数

任务状态

在 celery 中我们可以获取当前任务的任务状态，Celery 其内建任务状态有如下几种：

参数 说明
PENDING 任务等待中
STARTED 任务已开始
SUCCESS 任务执行成功
FAILURE 任务执行失败
RETRY 任务将被重试
REVOKED 任务取消

我们还可以自建任务状态，只需要在 self.update_state 中 self.update_state(state=”PROGRESS”, meta={‘p’: i}) 设置 state 字段就可以，其中 meta 是返回的 result 值

我们还可以通过 AsyncResult 对象的 state 字段来获取当前任务状态

任务终止

如果你想终止 celery 中正在执行的任务，需要先根据 task_id 找到任务，然后调用 revoke 函数废弃任务（这里可能有个问题，当前这个 task_id 的任务我没法从 celery 队列中移除，还在尝试中，forget？）

我们通常监控 celery 任务状态用的是 flower ，实际上 flower 也只是通过调用命令行的方式去 celery 中查询任务状态啊，我们也可以直接用命令行调用 api 的方式来获取这些任务状态

补充

1.实际生产环境中推荐使用 config_from_object 作为配置文件加载方式，并把配置文件，配置文件加载，普通任务注册，定时任务注册，worker，customer 分隔开，放到不同的文件

2.celery（4.x以下测试）有个地方很坑，他的状态机机制有一点问题，只有当任务丢到队列并执行的时候才算 start，pending状态实际上包含了两种情况，一是任务没添加进来，二是任务添加进来后由于队列繁忙，任务被阻塞等待，这种情况下依然是 pending 状态

3.如果你的 celery 是基于 redis 来用的，他对消息队列状态控制是有问题的，如果进程被 kill -9 ，redis 里面会存储一个错误的状态，并且没法更新，很坑的。。

4.celery 异步任务结束他的返回值会作为 result 写到异步任务状态的元数据中，然后把异步任务状态元数据中的 result 置为 null
